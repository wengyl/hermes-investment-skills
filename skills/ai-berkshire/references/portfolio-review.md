# 组合管理：从"研究公司"到"管理组合"

对 $ARGUMENTS 执行投资组合审视与优化。

**支持输入格式**：
- 持仓清单，例如：`腾讯30%, 美团20%, 茅台20%, 英伟达15%, 现金15%`
- 或：`腾讯 500股 @480港元, 美团 1000股 @130港元, ...`
- 或：`我的持仓`（如果已有保存的组合文件 `reports/portfolio-latest.md`）

> "分散投资是对无知的保护。如果你知道自己在做什么，分散投资就没有意义。" —— 巴菲特
>
> "我这辈子见过的真正好的投资机会，十个手指就数得完。" —— 李录

## 设计理念

研究公司只是投资的一半。另一半是**组合层面的决策**：
- 买多少？（仓位）
- 用什么钱买？（资金来源——新钱还是换仓）
- 和已有持仓是否冲突？（相关性）
- 最优组合长什么样？（机会成本）

巴菲特从不孤立地看一只股票——他总是在想"这是不是我能做的最好的事？"

## 执行流程

### 第一步：解析持仓

从输入中解析出当前持仓，标准化为以下格式：

| 标的 | 代码 | 持仓量 | 成本价 | 现价 | 市值 | 占比 | 盈亏 |
|------|------|--------|-------|------|------|------|------|

如果输入只有比例没有金额，按比例分析即可。

同时检查是否存在已有的组合文件（`reports/portfolio-latest.md`），如有则读取并更新。

### 第二步：获取最新数据

使用 Task 工具启动后台 Agent，通过 WebSearch 为每个持仓并行获取：
1. 当前股价和估值指标（PE、PB、股息率）
2. 最近一个季度的关键财务变化
3. 近期重大事件
4. 分析师一致预期（前瞻PE、目标价）

对每个持仓使用 `tools/financial_rigor.py verify-valuation` 校验估值数据。对每只持仓标注信息丰富度（A/B/C级），C级持仓的分析结论标注低置信度。

### 第三步：单仓位体检

对每个持仓进行快速健康检查：

| 标的 | 当前PE | 买入逻辑是否变化 | 论文健康度 | 仓位建议 |
|------|:------:|:--------------:|:---------:|---------|
| 腾讯 | 18x | 未变化 | 8/10 | 合理 |
| 美团 | 25x | 竞争加剧 | 6/10 | 偏高，考虑减仓 |

对每个持仓回答：
- [ ] **如果今天没有持仓，你还会在当前价格买入吗？**
- [ ] **如果明天不能交易，持有5年你舒服吗？**
- [ ] **买入论文还完整吗？**

**段永平**："如果你不想持有一只股票10年，那就一天也不要持有。"

### 第四步：组合层面分析

#### 4.1 集中度分析

| 指标 | 当前值 | 建议范围 | 判断 |
|------|-------|---------|------|
| 第一大持仓占比 | | <40% | |
| 前三大持仓占比 | | 50-80% | |
| 总持仓数量 | | 5-15只 | |
| 现金占比 | | 10-30%（视市场环境） | |

**李录的标准**：3-5只核心持仓，前3占80%+。**但这要求每一只都研究透彻。**

**巴菲特的标准**：核心持仓不超过10只，但允许更多卫星仓位。

#### 4.2 相关性检查

识别持仓之间的隐性关联：

| 持仓A | 持仓B | 相关类型 | 风险 |
|-------|-------|---------|------|
| 腾讯 | 快手 | 同属中国互联网 | 监管风险共振 |
| 英伟达 | 台积电 | AI供应链上下游 | AI Capex同向波动 |
| 美团 | 拼多多 | 同属中国消费 | 宏观消费同向波动 |

**检查清单**：
- [ ] 是否有超过50%的仓位暴露在同一个主题/行业？
- [ ] 是否有超过50%的仓位暴露在同一个国家/货币？
- [ ] 如果中美关系恶化，组合会亏多少？
- [ ] 如果全球经济衰退，组合会亏多少？

#### 4.3 机会成本分析

这是巴菲特最核心的思维方式——**每一块钱都应该放在回报最高的地方**。

将所有持仓按"预期年化回报"排序：

| 排名 | 标的 | 当前占比 | 预期年化回报 | 确定性 | 预期回报×确定性 |
|:----:|------|:-------:|:----------:|:------:|:--------------:|
| 1 | | | | | |
| 2 | | | | | |
| ... | | | | | |

预期回报估算方法（使用 `tools/financial_rigor.py three-scenario` 计算）：
- **简化公式**：预期年化 ≈ FCF Yield + 预期增速（主要方法）
- **价值型验证**：安全边际回归 + 利润增速 + 股息率
- **成长型验证**：利润增速 × 合理PE的变化

**关键问题**：排名最后的持仓，预期回报是否高于现金（无风险利率~4%）？如果不是，应该卖出换成现金。

#### 4.4 压力测试

| 情景 | 假设 | 组合预计影响 | 最大回撤 |
|------|------|-----------|---------|
| 全球衰退 | 企业盈利下降20-30% | | |
| 中美冲突升级 | 中概股折价50% | | |
| 利率飙升 | 10年期国债→6% | | |
| 科技泡沫破裂 | 科技股PE压缩40% | | |

对每个情景做定性+粗估评估（基于各持仓的行业属性和历史估值波动范围）：
- 哪些持仓受冲击最大？大致影响方向和量级范围
- 组合整体是否能承受？
- 是否需要对冲？

### 第五步：优化建议

#### 5.1 调仓建议

基于以上分析，给出具体的调仓建议：

| 动作 | 标的 | 当前占比 | 建议占比 | 理由 |
|------|------|:-------:|:-------:|------|
| 加仓 | | | | |
| 减仓 | | | | |
| 清仓 | | | | |
| 新建仓 | | | | |
| 不动 | | | | |

#### 5.2 寻找替代标的

如果组合中有"不如现金"的仓位，或者现金占比过高，建议使用 `/industry-research` 或 `/investment-checklist` 对感兴趣的行业/公司进行系统筛选，而非在本Skill内直接推荐个股。

#### 5.3 现金管理

| 当前现金占比 | 建议现金占比 | 理由 |
|:----------:|:----------:|------|

**巴菲特**：目前持有$3,820亿现金，占比超过总资产的25%——当找不到好机会时，现金是最好的仓位。

### 第六步：输出组合报告

#### 报告结构

```
一、组合概览（持仓表格+饼图描述）
二、单仓位体检（每个持仓的健康状态）
三、组合分析
   - 集中度：是否过度分散/集中？
   - 相关性：隐性关联和风险共振
   - 机会成本：排名最低的仓位是否值得持有？
   - 压力测试：极端情景下的回撤预估
四、调仓建议（具体操作+理由）
五、下次审视时间和关注重点
```

#### 结论必须明确回答

1. **组合整体健康度**：优秀 / 良好 / 需要调整 / 问题严重
2. **最应该做的一件事是什么？**（加仓X / 减仓Y / 不动）
3. **当前最大风险是什么？**

### 第七步：保存组合文件

将组合信息写入 `reports/portfolio-latest.md`，包含：
- 最新持仓表
- 本次审视日期和结论
- 调仓记录（追加）
- 下次审视提醒

## 关键原则

- **每一块钱都有机会成本** — 持有一只平庸的股票，成本是错过了一只优秀的
- **集中不是风险，无知才是** — 持有3只你深度理解的股票，比持有30只你一知半解的安全
- **现金是一种仓位** — 找不到好机会时，持有现金不丢人
- **组合层面 > 个股层面** — 一只好股票在错误的仓位上也会拖累你
- **定期审视，但不要过度交易** — 每季度审视一次足够，不要每天盯盘调仓

## 基金组合审计框架（Hermes 适配）

当用户使用 Hermes fund-investment-advisor 系统管理基金组合时，ai-berkshire 的组合审视框架可用于审计自动化简报系统的质量。

### 审计入口

用户说"分析一下投资简报哪些需要优化"或"审计投顾系统"时：

1. **读取系统当前输出**：运行 `python3 scripts/advisor.py morning` 获取完整简报
2. **读取历史报告**：检查 `reports/weekly_review_latest.json` 和 `reports/monthly_evaluation_latest.json`
3. **读取cron配置**：`hermes cron list` 确认定投/简报cron状态
4. **读取数据库状态**：`holdings` 表基金列表 vs 简报显示的基金列表
5. **逐模块审计**：按 P0/P1/P2 优先级分类问题

### 审计维度（四大师视角）

| 大师 | 审计追问 | 对应系统检查 |
|------|---------|-------------|
| 芒格 | "这个系统最大的问题是什么？" | 数据准确性（P0级） |
| 巴菲特 | "输入的是垃圾还是黄金？" | 数据源是否对齐DB |
| 段永平 | "这个功能好在哪？" | 简报是否真正可操作 |
| 李录 | "站在3年后回看，哪些是核心？" | 风控逻辑/估值分位/基准对比 |

### 常见问题清单

- **持仓数据不同步**：简报显示已清仓基金 → 检查 `WHERE share_count > 0` 过滤
- **定投信息过期**：DCA已切换但简报仍显示旧基金 → 检查 `dca_plans` 表和 cron
- **风控评分矛盾**：新建仓基金被建议"减仓" → 检查新建仓豁免逻辑（<90天）
- **均线信号混淆**：模拟信号与实际操作建议格式相同 → 加"模拟信号"标注
- **月度评估全零**：虚拟回测无交易 → 加真实持仓NAV加权收益评估

详见 `fund-investment-advisor` skill 的 `references/pitfalls.md` 条目 32-46 和 `references/briefing-audit-fixes.md`。

### 审计后验证清单

审计修复后必须逐项验证：

1. **持仓完整性**：`python3 scripts/advisor.py morning 2>/dev/null | grep -E "^[0-9]{6}"` - 确认所有基金显示
2. **旧基金清除**：`grep -c "OLD_CODE" /tmp/morning_report.txt` - 应为0
3. **新基金收益**：确认新基金有非零今日收益（需先 backfill share_count）
4. **定投信息**：DCA信息从DB动态读取，不出现旧基金定投
5. **风控豁免**：新建仓基金（<90天）显示"新建仓观察期"而非"考虑减仓"
6. **移动止盈措辞**：显示"分批止盈信号"而非"移动止盈"
7. **均线信号标注**：显示"模拟信号，非实际操作建议"
8. **基准对比**：显示组合收益 vs 沪深300
9. **估值分位**：PB>95%的基金有🔴标记
10. **重叠度操作建议**：严重重叠有个股有"减仓1/3"建议
11. **组合穿透**：行业配置末尾有"组合穿透"汇总表
12. **风控矛盾消除**：仓位>20%的基金不出现"增持"建议

### 加减仓逻辑审计框架（found 2026-07-08）

当用户说"分析一下投资播报的加减仓逻辑是否合理"时，使用此框架审计三套决策引擎的一致性。

**三套引擎架构**：
```
引擎1  策略建议(advisor.py)           多维度打分   -> "可加仓/持有观望/止盈减仓"
引擎2  风控评分(adaptive_risk_v2.py)  5因子加权    -> "持有观望/考虑减仓/建议止盈"
引擎3  场外策略(oof_strategy_advisor) 移动止盈+风险平价 -> "止盈信号/配置建议"
```

**审计维度（四大师视角）**：

| 大师 | 审计追问 | 对应系统检查 |
|------|---------|-------------|
| 芒格 | "三个顾问给你三个不同建议，你怎么做决策？" | 三引擎矛盾度（P0级） |
| 巴菲特 | "盈亏率是价值吗？" | "价值"维度实为盈亏率非估值（P0级） |
| 巴菲特 | "每一块钱是否放在回报最高的地方？" | 无机会成本分析（P2级） |
| 段永平 | "基于一天的资金流向就加仓？" | 加仓阈值过低，仅需2分（P1级） |
| 巴菲特 | "风险平价说增配，估值88%说减配，听谁的？" | 风险平价不考虑估值（P1级） |
| 段永平 | "信号每天都在响但你从不执行？" | 止盈信号无触发天数追踪（P3级） |

**核心审计清单（10项）**：

1. **P0-三引擎矛盾**：同一基金在策略建议/风控/场外模块收到矛盾指令（如"可加仓" vs "建议止盈"）
2. **P0-"价值"维度名不副实**：引擎1的"价值"策略视角实际使用盈亏率(profit_rate)，不是真正的估值分析
3. **P1-加减仓不考虑持仓重叠度**：5只高重叠基金全部建议"可加仓"，等于在同一个篮子放更多鸡蛋
4. **P1-风险平价可建议买入高估值基金**：引擎3基于波动率配置，PB分位88%的基金仍被建议增持
5. **P1-"可加仓"阈值过低**：仅需2分（行业受益热点+2，强于大盘+1），单日噪音即可触发
6. **P2-估值分位硬编码**：HIGH_PB_FUNDS字典手动维护，新基金无数据
7. **P2-定投建议是占位符**：_get_dca_suggestions()返回静态文本，不知道当前PE分位
8. **P2-无机会成本分析**：不回答"减仓后的钱应该买什么"
9. **P3-止盈信号只报不执行**：002112等连续触发多天但仓位不变，用户产生信号疲劳
10. **P3-集中度阈值与重叠度脱节**：CR4/CR8集中度分析不与持仓重叠度联动

**修复方案（已实现）**：

- **统一决策仲裁器**（`decision_arbiter.py`）：优先级 risk_stop > trailing_stop > valuation > overlap > trend。高优先级建议减仓时，趋势引擎的"可加仓"被否决为"持有观望"+原因。`batch_arbitrate()` 预加载所有引擎数据一次，避免N×DB查询。
- **"价值"重命名为"盈亏止盈"**：区分真正估值分析（PE/PB分位）与盈亏止盈逻辑。
- **加仓阈值2->3**：需要至少趋势+动量+一个其他因素同时满足。
- **估值分位动态计算**：从90日净值区间位置动态算，不再硬编码。
- **定投建议动态化**：接入沪深300指数90日区间位置近似PE分位。
- **机会成本分析模块**：按30天收益排序所有持仓，标注"可加仓/止盈减仓/清仓/考虑止损"。
- **止盈信号触发天数**：显示"已触发N天"+紧急度星标。

### 份额反推（screenshot sync 后必须执行）

截图同步后，`share_count=0` 的基金需要从 `current_value / latest_nav` 反推份额：

```python
import sqlite3
conn = sqlite3.connect('data/fund_system.db')
c = conn.cursor()
for code in ['013945', '014028', '025936']:  # share_count=0 的基金
    c.execute('SELECT unit_nav FROM fund_nav_history WHERE fund_code=? ORDER BY nav_date DESC LIMIT 1', (code,))
    nav = c.fetchone()[0]
    c.execute('SELECT current_value FROM holdings WHERE fund_code=?', (code,))
    cv = c.fetchone()[0]
    if nav and nav > 0 and cv and cv > 0:
        shares = cv / nav
        c.execute('UPDATE holdings SET share_count=? WHERE fund_code=?', (shares, code))
        # Also backfill avg_cost
        c.execute('SELECT total_invested FROM holdings WHERE fund_code=?', (code,))
        ti = c.fetchone()[0]
        if ti and ti > 0:
            c.execute('UPDATE holdings SET avg_cost=? WHERE fund_code=?', (ti/shares, code))
conn.commit()
conn.close()
```

不做这步，新基金的今日收益永远为0。
